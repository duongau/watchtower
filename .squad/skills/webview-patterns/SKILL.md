## Context

The CDA Extension uses a **WebviewView** (sidebar panel), not a standalone WebviewPanel.
The sidebar is registered as `cda.welcomeView` in `package.json` and provided by
`CdaSidebarProvider` (`src/panels/sidebarProvider.ts`).

Key architecture facts:
- HTML is generated server-side in `src/panels/wizard-html.ts` (two modes: `buildWizardHtml` for onboarding, `buildTabbedHtml` for post-wizard dashboard)
- Webviews use **vanilla JS** — no React, Vue, or framework. All DOM manipulation is imperative.
- Communication between extension host and webview uses VS Code's message-passing API.
- All inline scripts and styles require a **nonce** for CSP compliance.
- The nonce is generated by `getNonce()` in `src/panels/utils.ts` using `crypto.randomBytes(16)`.

## Patterns

### 1. Type-safe message passing

Every message between extension host and webview uses a `command` string discriminator
plus a payload. Define TypeScript interfaces for all messages.

**Extension host → webview (postMessage):**
```typescript
// Extension host sends data to webview
this._view.webview.postMessage({
  command: 'updateToolsStatus',
  tools: toolStatusArray,
  allInstalled: true
});
```

**Webview → extension host (vscode.postMessage):**
```javascript
// Inside webview <script nonce="${nonce}">
const vscode = acquireVsCodeApi();

// Send a message to the extension host
vscode.postMessage({
  command: 'saveProfile',
  alias: aliasValue,
  userName: nameValue,
  team: teamValue,
  role: roleValue
});
```

**Extension host receives messages:**
```typescript
// In resolveWebviewView()
webviewView.webview.onDidReceiveMessage(
  async (message) => this._handleMessage(message),
  null,
  this._disposables
);
```

**Webview receives messages:**
```javascript
// Inside webview script
window.addEventListener('message', function(event) {
  const msg = event.data;
  switch (msg.command) {
    case 'updateToolsStatus':
      renderTools(msg.tools);
      break;
    case 'viewBecameVisible':
      refreshUI();
      break;
  }
});
```

**CDA convention:** The message shape is always `{ command: string; [key: string]: unknown }`.
The `command` field is the discriminator. See `_handleMessage()` signature in sidebarProvider.ts.

### 2. Message handler — switch statement pattern

CDA uses a single `_handleMessage` method with a large `switch` on `message.command`.
Each case handles one message type. This is the established pattern — follow it for consistency.

```typescript
private async _handleMessage(message: { command: string; [key: string]: unknown }): Promise<void> {
    console.log('[CDA] webview message:', message.command);
    switch (message.command) {
        case 'saveProfile': {
            const alias = String(message.alias ?? '').trim().slice(0, MAX_FIELD_LEN);
            // ... validate and save
            break;
        }
        case 'requestToolsCheck': {
            const statuses = await this._checkTools();
            this._view?.webview.postMessage({ command: 'updateToolsStatus', tools: statuses });
            break;
        }
        // ... more cases
    }
}
```

**Key practices in CDA's handler:**
- Always sanitize inputs: `String(message.field ?? '').trim().slice(0, MAX_FIELD_LEN)`
- Use `escapeHtml()` from `src/panels/utils.ts` for any value going into HTML
- Log the command: `console.log('[CDA] webview message:', message.command);`
- Use block-scoped cases (`case 'x': { ... break; }`) to avoid variable hoisting issues

### 3. State persistence

CDA uses two layers of state persistence:

**Webview-side state** (`vscode.getState()` / `vscode.setState()`):
Survives webview hide/show but NOT extension host restarts.

```javascript
// In wizard-html.ts — restore state on load
const vscode = acquireVsCodeApi();
let savedState = vscode.getState();
let state = {
  currentStep: resumeStep,  // Server-side always wins
  alias: savedState?.alias || serverDefault,
  tools: savedState ? savedState.tools : {},
  // ...
};

// Save state whenever it changes
function saveState() {
  vscode.setState(state);
}
```

**Extension-host-side state** (`context.globalState`):
Survives restarts. Used for durable settings and wizard progress.

```typescript
// Read
const themeId = this._context.globalState.get<string>('cda.theme', DEFAULT_THEME);
const wizardStep = this._context.globalState.get<number>('wizardStep', 1);

// Write
await this._context.globalState.update('cda.wizardCompletedAt', new Date().toISOString());
await this._context.globalState.update('wizardStep', step);
```

**CDA convention:** Use `globalState` for anything that must survive extension restarts
(wizard step, theme, completed-at timestamp, pending installs, skill enabled/disabled flags).
Use webview `setState` for ephemeral UI state (form values in progress, scroll position).

**Namespace convention:** All globalState keys use `cda.` prefix (e.g., `cda.theme`, `cda.wizardCompletedAt`).
Legacy keys without prefix (e.g., `wizardStep`) exist but new keys must use `cda.*`.

### 4. Content Security Policy (CSP)

CDA uses strict CSP with nonce-based script and style loading. No `unsafe-inline`, no `unsafe-eval`.

```typescript
// In wizard-html.ts — the CSP meta tag
`<meta http-equiv="Content-Security-Policy"
  content="default-src 'none'; style-src 'nonce-${nonce}'; script-src 'nonce-${nonce}';">`
```

**Every `<script>` and `<style>` tag must include the nonce:**
```html
<style nonce="${nonce}">${CSS_CONTENT}</style>
<script nonce="${nonce}">
(function() {
  const vscode = acquireVsCodeApi();
  // ... all JS here
})();
</script>
```

**Nonce generation** (`src/panels/utils.ts`):
```typescript
export function getNonce(): string {
    return crypto.randomBytes(16).toString('base64');
}
```

**CDA convention:** Generate nonce once per `_refresh()` call. Pass it to both `buildWizardHtml(nonce, ...)` and `buildTabbedHtml(nonce, ...)`. The nonce is a parameter to the HTML builder functions, not a global.

### 5. Resource URI resolution

Use `webview.asWebviewUri()` for any local file (scripts, CSS, images) referenced in webview HTML.

```typescript
// In resolveWebviewView — set allowed roots
webviewView.webview.options = {
    enableScripts: true,
    localResourceRoots: [
        vscode.Uri.joinPath(this._context.extensionUri, 'media'),
    ],
};

// Resolve a URI for the webview
const scriptUri = webviewView.webview.asWebviewUri(
    vscode.Uri.joinPath(this._context.extensionUri, 'media', 'main.js')
);
```

**CDA convention:** CDA currently inlines all CSS and JS directly in the HTML string
(no external script files). If you add external resources, they must go in `media/`
and be referenced via `asWebviewUri()`. The `localResourceRoots` is already set to `media/`.

### 6. Webview lifecycle

The sidebar webview can be hidden and reshown — handle lifecycle events properly.

```typescript
// In resolveWebviewView()
webviewView.onDidChangeVisibility(() => {
    if (webviewView.visible) {
        // Notify webview it's visible again
        this._view?.webview.postMessage({ command: 'viewBecameVisible' });
    }
}, null, this._disposables);
```

**CDA convention:**
- `resolveWebviewView()` is called once when the sidebar is first shown
- The `_refresh()` method rebuilds the entire HTML — it's called on initial load and when state changes
- `_view` may be undefined if the sidebar hasn't been opened yet — always null-check: `this._view?.webview`
- Disposables are tracked in `this._disposables` and cleaned up automatically
- Polling timers (e.g., for WI/PR status) start in `resolveWebviewView()` and respect `cda.pollInterval` setting

### 7. Auto-save field pattern

CDA uses `autoSaveField(elementId, settingKey)` to wire HTML form inputs to VS Code settings
with auto-persistence on change and blur.

```javascript
// Defined in wizard-html.ts webview script
function autoSaveField(elementId, settingKey) {
  var el = document.getElementById(elementId);
  if (el) {
    el.addEventListener('change', function() {
      vscode.postMessage({command:'settingToggle', key: settingKey, value: el.value});
    });
    // Also save on blur for text inputs
    if (el.tagName === 'INPUT') {
      el.addEventListener('blur', function() {
        vscode.postMessage({command:'settingToggle', key: settingKey, value: el.value});
      });
    }
  }
}

// Usage — wire each form field to its setting key
autoSaveField('settAuthorName', 'authorName');
autoSaveField('settAuthorEmail', 'authorEmail');
autoSaveField('settTeamService', 'teamService');
autoSaveField('settInstallLocation', 'installLocation');
autoSaveField('settSkillsDir', 'skillsLocations');
autoSaveField('settPollInterval', 'pollInterval');
```

**How it works:**
1. HTML input has an `id` matching the first argument (e.g., `id="settAuthorName"`)
2. On `change` (selects) or `blur` (text inputs), the value is sent to extension host
3. Extension host receives `settingToggle` command and writes to `cda.*` VS Code settings
4. The setting key maps to `cda.{settingKey}` (e.g., `authorName` → `cda.authorName`)

**When to use:** Any new settings field in the sidebar should use `autoSaveField`.
The HTML `id` must match the `elementId` argument. The `settingKey` must match a
setting declared in `package.json` under `contributes.configuration`.

## Anti-Patterns

- **Never use `eval()` or `innerHTML` with untrusted content.** Use `escapeHtml()` from `src/panels/utils.ts` for all dynamic text. CDA already does this consistently.
- **Never fetch remote scripts in webviews.** CSP blocks it (`default-src 'none'`). All code must be inline with nonce or loaded from `localResourceRoots`.
- **Never hardcode file paths.** Always use `asWebviewUri()` for local resources. CDA inlines everything, but if you externalize, use the URI helper.
- **Never put business logic in the webview.** Keep validation, file I/O, API calls, and state management in the extension host. The webview is a thin UI layer that sends messages.
- **Never use `unsafe-eval` or `unsafe-inline` in CSP.** CDA uses nonce-based CSP exclusively. Adding `unsafe-*` defeats the security model.
- **Never use inline `onclick` handlers.** CDA uses `addEventListener` exclusively (CSP-safe). See the `// --- Event listeners (CSP-safe, replaces inline onclick) ---` section in wizard-html.ts.
- **Never assume `this._view` is defined.** The sidebar may not be open. Always use `this._view?.webview`.
